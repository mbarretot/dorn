# eng/

Engineering scripts and tooling that support the Dorn repository itself ‚Äî as opposed to
`src/`, `templates/`, and `tests/`, which are the product. Nothing under `eng/` ships to
end users of `dorn` or of a generated project.

## `packages/` and `scripts/pack-packages.ps1`

`packages/` (a top-level directory, sibling to `src/`, `templates/`, `tests/`) holds
Dorn's own first-party library packages that generated projects depend on at runtime:

- `Dorn.Messaging.Contracts` ‚Äî pure mediator interfaces (`IRequest`, `ISender`,
  `INotification`, etc.), zero dependencies.
- `Dorn.Messaging` ‚Äî the mediator implementation (`Mediator`, `AddMediator`).
- `Dorn.SharedKernel` ‚Äî DDD building blocks (`Entity`, `AggregateRoot`, `Result`).

`templates/webapi` consumes all three via ordinary `PackageReference`, versioned in its
own `Directory.Packages.props`. None of them are published to NuGet.org yet, so
`eng/scripts/pack-packages.ps1` packs all three into `./artifacts`, which the root
`nuget.config`'s `dorn-local` source resolves as a local package feed. Run it before
`dotnet restore Dorn.sln` ‚Äî otherwise `templates/webapi`'s restore fails to find these
packages:

```bash
pwsh eng/scripts/pack-packages.ps1
pwsh eng/scripts/pack-packages.ps1 -Version 1.2.3
```

See `docs/adr/0011-extract-messaging-and-shared-kernel-as-nuget-packages.md` for the full
decision record, including why the code moved from the old physical-copy-and-diff-check
template-sync convention (ADR 0008, now superseded) into real packages.

## `scripts/pack-templates.ps1`

Packages `templates/webapi` as a standalone NuGet template package
(`<PackageType>Template</PackageType>`), installable via vanilla
`dotnet new install Dorn.Templates.WebApi` ‚Äî independent of the `dorn` CLI tool, and
discoverable in Visual Studio's "Create a new project" search via the same
`PackageType=Template` mechanism. This is the "dual distribution" path: `templates/webapi`
stays installable through the `dorn` CLI's own embedded Template Engine host (ADR 0002)
*and* through this NuGet package ‚Äî both point at the same `templates/webapi/` content,
so there's no drift risk between the two channels.

What it does:

1. Runs `dotnet pack` against `eng/packaging/Dorn.Templates.WebApi/Dorn.Templates.WebApi.csproj`
   ‚Äî a pack-only project **outside** `templates/webapi/`, so the packaging project itself
   never gets instantiated into a generated user project. It is not part of `Dorn.sln`;
   invoke it directly via `dotnet pack <path>` or through this script.
2. Emits the `.nupkg` to `./artifacts`, versioned via an optional `-Version` parameter
   (default `0.1.0-dev`).

```bash
pwsh eng/scripts/pack-templates.ps1
pwsh eng/scripts/pack-templates.ps1 -Version 1.2.3
```

See `docs/adr/0009-dual-distribution-dotnet-new-template-pack.md` for the full decision
record, including why this was originally deferred and later brought into v1 scope.
