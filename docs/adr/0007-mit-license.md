# 0007. MIT License

## Status

Accepted

## Context

Dorn is a community scaffolding tool intended to be freely adopted, extended, and
redistributed with minimal friction — both the CLI itself and every project it generates.
The choice of license affects who can adopt Dorn (permissive licenses lower the bar for
commercial and hobby use alike), how contributors relate to their contributions, and,
transitively, what license every project generated by `dorn new webapi` effectively
inherits from the template code it's built from.

This decision is also downstream of, and consistent with, other decisions already made
in this repo: MediatR's move to RPL-1.5/commercial licensing (ADR 0003) and
FluentAssertions's move to a commercial license (ADR 0006) were both explicitly avoided
specifically to keep every project Dorn generates free of licensing encumbrances or
friction. Choosing anything less permissive than MIT for Dorn's own license, or for the
template source it ships, would undercut that goal at the source.

## Decision

Dorn is licensed under the MIT License, applied at the repository root (`LICENSE`) and
covering the CLI tool (`src/`), the templates (`templates/`, including the code every
generated project starts from), and this documentation. Contributors agree their
contributions are licensed under the same terms (`docs/contributing.md`).

## Consequences

- Maximum permissiveness for adoption: individuals, companies, and other open-source
  projects can use, modify, and redistribute Dorn and Dorn-generated projects with
  effectively no restriction beyond preserving the license notice.
- Consistent with every other licensing-sensitive decision already made in this repo
  (ADR 0003, ADR 0006) — nothing in the dependency graph of a generated project
  introduces a licensing obligation MIT itself doesn't already cover.
- MIT provides no patent grant (unlike, e.g., Apache 2.0) — accepted as a minor
  trade-off; a project of this scope and community-driven nature was judged to value
  simplicity and maximum permissiveness over explicit patent protection.
- Future dependencies (new NuGet packages added to `Directory.Packages.props` in either
  the root or `templates/webapi`) should be vetted for license compatibility with MIT
  distribution before being added, following the same reasoning applied to MediatR and
  FluentAssertions.
